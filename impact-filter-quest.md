# Impact Filter: Quest

## 1. Selling Yourself

_Before communicating this to others, ensure you are fully committed to the idea._

- **Purpose:** A task dependency tool for agents swarming complex deliverables.
- **Importance:** This is an essential component for my agent orchestration system. My unique domain insights do not exist in any other tool. This could enable a massive productivity boost relative to other orchestrators.
- **Ideal Outcome:** Quest is extremely easy to use because it has a simple and minimal command set with every command and flag being clear in its intent and obvious in its usage. Agents working on quest tasks have exactly what they need readily at-hand, both to accomplish the task and give updates on progress. Agents decomposing large deliverables and creating tasks can directly express their intentions with tool calls to quest. They can create dependencies and relationships between tasks without ever getting confused. Quest can display a task graph that makes it easy to verify the correctness of what was captured, both for agents and humans.

## 2. Best & Worst Results

_Clarify the stakes by exploring the extremes._

- **Best Result:** My framework is so effective that it gains popularity and notoriety, opening all kinds of doors for me. I built one of the first genuinely self-improving tools. I unlock a new level of massive productivity for the world.
- **Worst Result:** We get stuck using task tools that are bloated, complex, and have too many features and dependencies that don't align well with the orchestration concepts of the framework. The framework is cumbersome to use and never becomes popular. Other builders with worse ideas thrive while my tool languishes. I miss out on the life-changing opportunities that arise from shipping innovation at a pivotal moment.

## 3. Success Criteria

_List the specific, measurable results that must be true for this to be a success._

- [ ] Quest can be installed by a new user and used on a project by typing 5 or fewer commands on Linux, Mac, and Windows
- [ ] When an agent is instructed to track work in quest, it uses quest commands for all task lifecycle actions (accept, update, complete, fail) rather than falling back to built-in task tracking or unstructured status reporting
- [ ] When agents interact with quest, fewer than 1% of commands result in a structural error (wrong dependency direction, wrong relationship type, task created under wrong parent, incorrect status transition) as measured by session log review
- [ ] All changes to a task leave an audit trace of who, what, and when
- [ ] Every command and flag maps directly to a clear and necessary use case
- [ ] The readme gets a power user (non-developer) up and running with it within 5 minutes
- [ ] All quest data can be exported to human-readable files that can be inspected, backed up, and version-controlled without quest-specific tooling
