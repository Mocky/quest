# Changelog

## [Unreleased]

### Added

- `commits` field on the task schema and `--commit BRANCH@HASH` flag on `quest update`, `quest complete`, and `quest fail` -- parallel to the existing `prs`/`--pr` surface, for work that lands via direct commit rather than (or in addition to) a PR. The flag is repeatable; no standalone `quest commit` command is added. Both halves of the value are required (`--commit master@` and `--commit @abc123` return exit 2 naming the flag); the parser splits on the rightmost `@` so branch names containing `@` (`release@2025`) parse correctly. Hashes must be lowercase hex and at least 4 characters; quest performs no git-awareness beyond the shape check. Dedup for idempotency is byte-equal after lowercasing the hash with branch preserved case-sensitive -- duplicates are silently ignored and produce no history entry, matching the `--pr` model. Terminal-state gating mirrors `--pr`: accepted on `completed`/`failed` tasks for late-discovered refs, rejected on `cancelled`. A new `commit_added` history action is appended on every successful add, carrying `branch` and `hash` as separate fields (not a combined ref string) so retrospective queries can filter on either half without re-parsing. `quest show` emits `"commits": []` in JSON (after `prs`) and renders a `Commits` section in text mode (one row per commit as `{branch}@{hash}  ({added_at timestamp})`), parallel to the existing `PRs` section. Spec-only; implementation (schema migration, storage table, command wiring, history row) ships in the corresponding impl task. Pinned in spec §Task Entity Schema, §History field, §quest update, §quest complete, §quest fail, §Commit reference format, §Idempotency, §quest show. `STANDARDS.md` §CLI Output Contract Tests adds `commits` to the `TestShowJSONHasRequiredFields` list in lockstep.
- Optional `severity` field on the task schema -- enum `critical`, `high`, `medium`, `low`; null when unset; case-sensitive lowercase. Applies to every task regardless of `type`. Set via `quest create --severity VALUE`, the `severity` key in a `quest batch` line, and `quest update --severity VALUE` (elevated-only, same terminal-state gating as other planning fields). `quest list --severity VALUE` filters by severity (CSV within the flag for OR, repeatable flag to accumulate, AND across dimensions with other filters); null-severity tasks are excluded from filter matches, consistent with `--role`. `severity` is a valid `--columns` value on `quest list` but is not in the default column set -- callers doing triage opt in explicitly. `quest show` emits `severity` in JSON (after `tier`, `null` when unset) and in text mode renders a `severity` row in the metadata cluster when non-null. `quest graph --text` does not render severity in the tree (JSON nodes carry it). Invalid values are rejected with exit code 2 on `create`/`update`/`list` and with a semantic-phase `invalid_severity` line in `quest batch` output. Existing title-prefix conventions (`CRITICAL: ...`) are not back-filled -- setting severity via the new flag is additive; agent playbooks that previously relied on title prefixes should be updated separately to use `--severity`. Spec-only; implementation (schema migration, CHECK constraint, command wiring) ships in the corresponding impl task. Pinned in spec §Task Entity Schema, §quest create, §quest update, §quest batch, §quest list, §quest show, §quest graph.
- `quest list --text` now renders a count footer after the table: a blank line, then `N tasks` (or `1 task` when exactly one, `0 tasks` when the result set is empty). Text-mode only; JSON output is unchanged because agents already get the count from the array length and adding a field would break the existing contract for zero agent benefit. Pinned in spec §quest list.
- Enforce the 128-byte cap on task `title` at every write path per spec §Field constraints. `quest create --title` and `quest update --title` exit 2 with a usage error naming the flag and the observed byte count; `quest batch` reports over-limit titles as a per-line `field_too_long` JSONL error in the `semantic` phase, carrying `field`, `limit`, and `observed`. Bytes, not code points — consistent with the `@file` 1 MiB limit.
- `quest backup --to PATH` writes a transaction-consistent snapshot of `.quest/quest.db` via SQLite's online backup API, plus a sidecar copy of `.quest/config.toml` at `PATH.config.toml`. Safe to run while agents operate on the workspace. Elevated-role-gated. See spec §Backup & Recovery.
- Automatic pre-migration snapshot: when the binary runs pending schema migrations, it first writes `.quest/backups/pre-v{N}-{timestamp}.db` via SQLite's online backup API. Snapshot failure aborts the migration (exit 1). See spec §Storage > Pre-migration snapshot.

### Changed

- Terminal status value renamed from `complete` to `completed` to match the past-participle pattern of the other terminal statuses. The CLI command `quest complete`, the `tx_kind=complete` telemetry attribute, and log event `"quest command complete"` are unchanged. Existing `.quest/quest.db` files migrate automatically on next invocation via schema v2.
- Session-ownership enforcement on `quest update` / `complete` / `fail` is now controlled by `enforce_session_ownership` in `.quest/config.toml` and defaults to `false` (spec §Role Gating > Session ownership). Previously the check was unconditional after `quest accept`; on upgrade, projects that omit the field load with enforcement off and no longer need `AGENT_SESSION` coordination between cooperating callers. Operators who want the old strict behavior set `enforce_session_ownership = true`. `owner_session`, `handoff_session`, and history `session` are recorded identically in both modes, so telemetry and retrospectives are unaffected.
- Role gating is now opt-in restriction rather than opt-in elevation (spec §Role Gating > Resolution order). An unset `AGENT_ROLE` gets the full command surface; an explicit `AGENT_ROLE` in `elevated_roles` also gets the full surface; an explicit `AGENT_ROLE` *not* in `elevated_roles` is the only state that reduces the caller to worker commands and returns exit 6 on anything else. Previously an unset role defaulted to the worker surface, which forced humans and non-vigil callers to export `AGENT_ROLE=planner` to use quest at all, and invited dispatched agents to self-elevate. Vigil activates the gate by setting an explicit role on every dispatch; a dispatched worker cannot lift its own restriction because the explicit role is what enables the gate.
- Worker commands (`show`, `accept`, `update`, `complete`, `fail`) no longer fall back to `AGENT_TASK` when a task ID is omitted. The task ID is now a required positional argument; a missing ID returns exit 2 (`usage_error`). `AGENT_TASK` remains an identity/telemetry env var set by vigil (recorded as `dept.task.id` on spans and log records) but is not consumed as a CLI default. Rationale: letting identity env vars double as CLI conveniences invited agents to set their own `AGENT_TASK` / `AGENT_ROLE` / `AGENT_SESSION` when constructing commands, which undermines both role gating and session ownership as intended signals from the framework.
- Help-text rendering convention pinned in `STANDARDS.md` §Help Rendering: long flags render with `--`, short flags with `-`, matching how they appear in the spec, README, and command examples. Go stdlib's default `flag.PrintDefaults` prefixes every flag with a single dash regardless of name length, so `quest <cmd> --help` has rendered long flags as `-status` / `-columns` / `-ready` since initial release — contradicting every doc that writes them with `--`. Error messages from flag parsing (e.g., `flag provided but not defined: -text`) come from a separate stdlib code path and are explicitly deferred.
- `quest <cmd> --help` now renders long flag names with `--` and single-character flag names with `-`, implementing the convention pinned in `STANDARDS.md` §Help Rendering. Every subcommand's `FlagSet` is constructed via one shared helper in `internal/command/flagset.go` that installs a custom `Usage` function, so new commands inherit the convention without per-command boilerplate. The layout (two-space indent, tab-then-usage for short headers, indented continuation for long headers) is unchanged; only the dash prefix differs. Flag-parse error messages remain deferred per the spec.
- Spec §Text-mode formatting pinned to content-aware rendering rules for `quest list --text`: helper columns (every column except `title`) size to `max(header_label_width, longest_cell_value)`; the `title` column is allocated the remainder of the terminal width up to the 128-byte title cap on a TTY, and is unbounded when stdout is not a TTY or the terminal width is unknown; narrow terminals are allowed to overflow rather than truncate helper content; the final column of each row is emitted without trailing whitespace so copy-paste is clean. The prior language ("columns use fixed widths ... when piped, columns use fixed default widths") described an intent the implementation never reached -- widths shipped as a hardcoded map in `internal/command/list.go`. Text mode remains explicitly non-contractual per STANDARDS.md §CLI Surface Versioning; the rendering can evolve without a deprecation cycle. No behavior change in this entry -- the implementation task (making widths content-aware, dropping the trailing pad) is tracked separately.
- `quest list --text` now renders the content-aware widths pinned by the preceding spec update. Helper columns (id, status, blocked-by, etc.) size to `max(header_label_width, longest_cell_value)` across the rows being printed, replacing the hardcoded widths map. The title column receives the remainder of the terminal width on a TTY (clamped to 128), is unbounded when stdout is not a TTY, and is allowed to overflow on narrow terminals where helpers alone exceed the terminal width. Every row drops the trailing pad on its final column so copy-paste no longer picks up invisible whitespace. Terminal-width detection reuses the existing `internal/output.TerminalWidth` helper; no new dependency. Text mode is still explicitly non-contractual per STANDARDS.md §CLI Surface Versioning.
- **Breaking:** the global `--format json|text` flag is renamed to the boolean toggle `--text`. JSON (the agent contract) is the default; `--text` selects the human-friendly rendering. Pinned in `quest-spec.md`, `STANDARDS.md` §Flag Overrides, and `GROVE.md` shared conventions. Config field renamed in lockstep: `OutputConfig.Format` (string, default `"json"`) -> `OutputConfig.Text` (bool, default `false`) -- the type now reflects the boolean nature of the knob. No env-var equivalent is provided (not even `QUEST_TEXT`): Claude Code sessions and other agents inherit shell env across many terminals, and a persistent default would silently corrupt agent output. Framework-wide change coordinated with lore so the `--format` convention does not diverge in quest alone; see grove `decisions.md` entry dated 2026-04-21.
- `--help` convention pinned in `STANDARDS.md` §`--help` Convention: every subcommand's flag set supports `--help`, which prints usage to stderr, exits 0, and short-circuits the command's work -- queries, writes, and dispatches do not run when `--help` is present. `--help` takes precedence over all other flags (`quest list --status cancelled --help` is equivalent to `quest list --help`). The invariant was previously implicit and inconsistently implemented; `quest list --help` surfaced the gap by emitting `[{},{},...]` on stdout after the usage text because `parseListFlags` swallowed `flag.ErrHelp` and the caller continued. Paired anti-pattern in `STANDARDS.md` §Anti-Patterns #7 calls out the flag-parse-helper shape that drops the `flag.ErrHelp` signal. Spec-only clarification; the `quest list --help` fix ships in the corresponding impl task.

### Deprecated
### Removed
- **Breaking:** `--format json|text` global flag. Replaced by the `--text` boolean toggle -- see the preceding Changed entry. Clean break with no deprecation alias: agents calling `quest <cmd> --format json` will get `flag provided but not defined: -format` (exit 2). Agents were always passing `--format json` to get the default, so the migration is to simply drop the flag; agents that want text mode swap `--format text` for `--text`.

### Fixed

- `quest list --help` no longer emits empty JSON objects to stdout after printing usage. The helper `parseListFlags` was swallowing `flag.ErrHelp` and returning a nil error, so `List` continued past the parse, ran the store query with the zero-value filter, and rendered one `{}` per task in the DB. The helper now propagates `flag.ErrHelp` and `List` short-circuits before querying, matching the `--help` convention pinned in `STANDARDS.md`. Audit of every flag-parse helper surfaced the same bug in `complete`, `fail`, `reset`, `move`, `cancel`, `update`, `link`, `unlink`, `create`, and `batch`; the two destructive cases (`cancel TASK --help` actually cancelled the task; `update TASK --help` wrote a DB update) are the reason this fix is broader than list. `link` and `unlink` were additionally restructured so flag parsing runs before the leading-positional check, which is what makes `quest link --help` (no TASK) print usage instead of failing with "task ID required". Regression test seeded a task and asserts `quest list --help` and `quest list --status cancelled --help` both emit empty stdout and exit 0.

### Schema

- Schema v3: added `CHECK` constraints on `tasks.status`, `tasks.type`, `dependencies.link_type`, and `history.action`. Previously the valid enum set was enforced only by Go-level guards in command handlers, so a future code path that bypassed them -- or any direct `sqlite3` write -- could plant an invalid value (the `complete` vs `completed` drift fixed by schema v2 is the precedent). Migrations now run on a dedicated connection with `PRAGMA foreign_keys=OFF` so the documented CREATE/INSERT/DROP/RENAME recreation pattern can retire the old tables; `PRAGMA foreign_key_check` audits integrity inside the transaction before commit, preserving the forward-only-never-partial contract. Existing `.quest/quest.db` files migrate automatically on next invocation.
- Schema v4: added nullable `severity` column on `tasks` with `CHECK (severity IS NULL OR severity IN ('critical','high','medium','low'))`. Forward-only migration using the same CREATE/INSERT/DROP/RENAME pattern as v3; existing rows land with `severity = NULL`. No retroactive backfill from title prefixes. Existing `.quest/quest.db` files migrate automatically on next invocation; the pre-migration snapshot fires as usual.
- Schema v5: added a new `commits` side table (`task_id` FK, `branch`, `hash`, `added_at`) with a `UNIQUE` index on `(task_id, branch, lower(hash))` for case-insensitive dedup; widened `history.action`'s CHECK constraint to include `commit_added`. Forward-only migration using the same table-recreation pattern for `history` as v3. Existing rows are unaffected; new tasks land with an empty commits array. Existing `.quest/quest.db` files migrate automatically on next invocation; the pre-migration snapshot fires as usual.

## [v0.1.0] - 2026-04-19

Initial release. Implements the v4 behavioral contract in [`docs/quest-spec.md`](docs/quest-spec.md).

### Added

- Workspace discovery by walking up from CWD to find `.quest/`, with `quest init --prefix` to create it.
- Worker commands: `quest show`, `accept`, `update`, `complete`, `fail`. `AGENT_TASK` defaults the target ID.
- Planner commands (gated by `AGENT_ROLE` against `elevated_roles`): `create`, `batch`, `cancel`, `reset`, `move`, `link`, `unlink`, `tag`, `untag`, `deps`, `list`, `graph`, `export`.
- System commands: `quest version`, `quest init`.
- Typed relationships — parent/child, `blocked-by`, `caused-by`, `discovered-from`, `retry-of` — with cycle and depth validation.
- `quest batch` accepts a JSON file of creates and edges; `--partial-ok` lands the lines that validated when others fail.
- Append-only history (`action`, `role`, `session`, `timestamp`, payload) written on every state-changing mutation; `quest show --history` surfaces it.
- `@file` and `@-` (stdin) input on every free-form flag (`--debrief`, `--description`, `--context`, `--note`, `--handoff`, `--reason`, `--acceptance-criteria`); 1 MiB cap; single-use `@-` per invocation.
- Structured JSON output on stdout per spec §Write-command output shapes; human-oriented text rendering selectable via a global flag (renamed to `--text` later -- see [Unreleased]).
- Stable exit codes (0/1/2/3/4/5/6/7) mapped from structured error sentinels; deterministic error precedence (role → existence → permission → state → usage).
- Serialized writes via `BEGIN IMMEDIATE` with a 5 s busy timeout; exit code 7 on contention (no internal retry — caller decides).
- Slog diagnostic logging on stderr with canonical event messages and attribute sets per `docs/OBSERVABILITY.md`.
- OpenTelemetry instrumentation (spans, metrics, slog bridge). Activates when `OTEL_EXPORTER_OTLP_ENDPOINT` is set; zero-cost no-op when unset. `OTEL_GENAI_CAPTURE_CONTENT` opts into content attributes.
- `quest export` writes a snapshot of per-task JSON, markdown debriefs, and JSONL history to `<workspace>/quest-export/` (sibling of `.quest/`, or `--dir`).
- Test suite: stdlib-only table-driven tests, Layer 2 contract tests (output shapes, history, precondition events), Layer 3/4 integration tests, Layer 5 race/concurrency tests.
- GitHub Actions CI running `make ci` (race + integration tags + coverage) on every push.

### Schema

- Initial schema (`schema_version = 1`) with `tasks`, `history`, `dependencies`, `tags`, `notes`, `prs`, `meta` tables. Migrations are forward-only; a newer-than-supported schema version causes the binary to refuse to run (exit 1).
