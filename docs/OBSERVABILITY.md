# Observability Standard

This document defines logging, error-handling, and output-contract conventions for quest. It serves two purposes: guiding human developers and instructing coding agents that work on this codebase.

All code — whether written by a human or generated by an agent — must follow these conventions. If you are a coding agent, treat every MUST/MUST NOT in this document as a hard constraint.

This doc is deliberately thinner than lore's equivalent. Quest's OpenTelemetry instrumentation (span inventory, metric definitions, attribute schemas, SDK wiring) lives in `OTEL.md` and is not repeated here. When this doc and `OTEL.md` overlap, `OTEL.md` is authoritative for telemetry. This doc owns:

- slog setup and logging conventions
- stdout / stderr output contract
- Error wrapping and sanitization
- Correlation identifiers that flow independently of OTEL

---

## Architecture Overview

Quest is a single-binary CLI. Every invocation is a short-lived process that:

1. Walks up from CWD to locate `.quest/`.
2. Loads configuration (see STANDARDS.md Part 1).
3. Initializes slog (and OTEL if configured — see OTEL.md §7).
4. Parses the command and dispatches to a handler.
5. Opens the local SQLite database at `.quest/quest.db`.
6. Executes the command (possibly inside a `BEGIN IMMEDIATE` transaction).
7. Emits result JSON to stdout, diagnostics to stderr.
8. Flushes OTEL, closes the database, exits.

There is no daemon, no wire protocol, no network I/O inside the tool. A `questd` daemon is noted in the quest spec as deferred; if it ships, this doc and `OTEL.md` extend accordingly.

---

## Principles

1. **Agents are the primary CLI consumers.** Stdout is a machine interface. Design it with the same rigor as an API.
2. **Data and diagnostics are separate channels.** Structured command results go to **stdout**. Everything else (slog records, error explanations, progress hints) goes to **stderr**.
3. **Every invocation is correlatable.** Trace context (`TRACEPARENT`) and agent identity (`AGENT_ROLE`, `AGENT_TASK`, `AGENT_SESSION`) flow from the caller's environment into spans, history entries, and log records. Quest does not generate its own per-invocation correlation ID — the trace/session context already in the environment plays that role.
4. **Logs are structured.** All diagnostic output uses `log/slog` with key-value fields. No `fmt.Println`, `log.Println`, or bare `os.Stderr.Write` for diagnostic output.
5. **Silence is success.** On success, stdout carries the response and stderr carries nothing unless the log level demands it. No banners, no confirmation prose, no decoration.
6. **Errors are actionable.** The stderr error message explains what went wrong and how to recover. The exit code (1-7) plus the structured JSON on stdout carry the machine-readable signal.

## Logger Setup

Use `log/slog` from the standard library. Do not introduce `zap`, `zerolog`, `logrus`, or any third-party logger without explicit justification.

### Initialization

Initialize the logger once at startup through the shared `internal/logging` package. A single invocation wires two handlers behind a fan-out:

```go
// In cmd/quest/main.go (roughly)
stderrH := logging.NewStderrHandler(logging.StderrConfig{
    Level:  logging.LevelFromString(cfg.Log.Level),   // default "warn"
    Format: logging.FormatText,                       // human-readable on stderr
    Output: os.Stderr,
})

otelH := logging.NewOTELHandler(logging.OTELConfig{
    Level: logging.LevelFromString(cfg.Log.OTELLevel), // default "info"
})

logger := slog.New(logging.NewFanoutHandler(stderrH, otelH))
slog.SetDefault(logger)
```

The two handlers are level-gated independently: stderr follows `QUEST_LOG_LEVEL`, the OTEL bridge follows `QUEST_LOG_OTEL_LEVEL`. This mirrors OTEL.md §3.1.

### Propagation

Pass the logger through context where possible. For deeply nested call chains that don't have context readily available (e.g., a small SQL helper), use the `slog.SetDefault` result — the default logger composes the same fan-out:

```go
// PREFERRED — logger accessed via context, trace/span IDs flow through automatically
func (h *Handler) handleAccept(ctx context.Context, cfg config.Config, taskID string) (protocol.Response, error) {
    slog.InfoContext(ctx, "handling accept", "command", "accept", "task.id", taskID)
    // ...
}

// ACCEPTABLE — for packages without direct context access
slog.Warn("sqlite busy_timeout exceeded, returning transient failure",
    "wait_ms", waitMS)

// WRONG — stdlib log, unstructured, goes to stderr without slog's handler chain
func handleAccept(...) {
    log.Println("handling accept") // NO
}
```

Always prefer the `*Context` variants (`InfoContext`, `WarnContext`, `ErrorContext`, `DebugContext`) inside request handlers. They are what the OTEL log bridge requires to capture trace and span IDs. Lifecycle calls that run before a span exists (SDK init, config loading errors in `main`) may use the non-context variants and accept that they will not carry trace correlation — this is correct; no trace exists yet.

## Correlation Identifiers

Quest does not mint its own per-invocation correlation ID. Three identifiers flow from the caller's environment and, together, correlate every quest action:

| Source env var  | Purpose                                                                               | Where it surfaces                                                         |
| --------------- | ------------------------------------------------------------------------------------- | ------------------------------------------------------------------------- |
| `TRACEPARENT`   | W3C trace context propagated by vigil. The root-span ID correlates quest with the caller. | Span attributes (see OTEL.md §4.3). Log records automatically carry `trace_id` and `span_id` via the `otelslog` bridge. |
| `AGENT_SESSION` | Unique session ID assigned by vigil.                                                  | Every history row (`sessions` field), span attribute `dept.session.id`, log field `dept.session.id`. |
| `AGENT_TASK`    | The agent's assigned task.                                                            | Default target for commands. Span attribute `dept.task.id`, log field `dept.task.id`. |

### Flow

```
Vigil starts agent session
  └─ Vigil sets AGENT_SESSION, AGENT_TASK, AGENT_ROLE, TRACEPARENT
       └─ Agent invokes quest
            └─ Quest reads env vars in internal/config/
                 └─ Quest creates root span, extracting TRACEPARENT
                      └─ Every span, log, and history row carries session and task IDs
                      └─ Every exit code maps 1:1 to a stable error class (OTEL.md §4.4)
```

### What Surfaces Where

- **Stdout** — never carries identifiers. Stdout is the command result: task JSON, batch ref→id mapping, graph structure. Identifiers are metadata, not data.
- **Stderr** — slog records include `dept.task.id` and `dept.session.id` when non-empty. `trace_id` and `span_id` are injected by a trace-enrichment wrapper around the stderr handler (`internal/logging/`): it calls `telemetry.TraceIDsFromContext` on the record's context and, when a span is active, adds the two IDs as slog attributes before the record reaches the text handler. When no span is active (telemetry disabled or lifecycle calls before a span exists), the fields are simply omitted. The OTEL log bridge (OTEL.md §3.1) is a separate, parallel handler for the OTEL log signal — it does not feed stderr.
- **History entries** — `role` and `session` fields on every mutation (quest-spec §History field). Recorded as `null` when the env var is unset.
- **Spans** — full attribute set per `OTEL.md §4.3`.

### When Environment Is Unset

Humans running `quest list` from a shell without vigil typically have none of these set. Quest behavior:

- `AGENT_ROLE` unset → role gating defaults to the worker surface (see spec Role Gating). In telemetry, recorded as the literal string `"unset"` per OTEL.md §8.6.
- `AGENT_TASK` unset → commands that require a task ID return exit 2 with a clear stderr message. Commands that accept a task ID as a positional argument proceed normally.
- `AGENT_SESSION` unset → history entries record `session: null`. Spans record the empty string.
- `TRACEPARENT` unset → quest creates a root trace for this invocation (OTEL.md §4.1).

Missing values are not an error; they surface as nulls in history and as empty/`"unset"` attributes in telemetry.

## Output Contract

### Stdout: Data Channel

- Every command writes its result to **stdout** in the format dictated by `--format` (default: `json`).
- On success, stdout is the only channel the caller needs to parse.
- On failure, stdout MAY be empty (e.g., a usage error caught before the handler runs) or MAY carry structured error content (e.g., a `quest batch` partial failure that still lists the created tasks and failed lines). The contract for which commands write structured data on failure is fixed in `quest-spec.md`.
- JSON output is compact, one object or array per logical response. JSONL (one JSON object per line) is used for streaming responses — currently `quest batch` output and `quest export history.jsonl`.
- Text-mode output (`--format text`) is a human-friendly rendering. **Agents MUST NOT parse text-mode output.** It is not a contract.

### Stderr: Diagnostic Channel

Everything that is not the command's structured result goes to stderr:

- slog records (per log-level configuration).
- Batch validation errors in `quest batch` (JSONL, one error object per line — this is a stderr stream, see spec Batch error output).
- Error explanations that accompany non-zero exit codes. Format: plain text, one or two lines, no slog framing:

  ```
  quest: conflict: proj-a1 has non-terminal children: proj-a1.1 (accepted), proj-a1.2 (open)
  quest: exit 5 (conflict)
  ```

- OTEL SDK internal errors (via `otel.SetErrorHandler`, see OTEL.md §7.4) are logged at WARN severity to stderr via slog, not printed directly.

The `quest` prefix on plain-text error lines is what distinguishes them from slog records (which have `level=WARN msg="..."` framing). Callers that grep stderr can match on `^quest: ` for the actionable error and ignore slog lines.

Example success invocation:

```
$ quest show proj-a1.3 --format json
# stderr: empty (log level defaults to warn; no warnings fired)
# stdout:
{"id": "proj-a1.3", "title": "Auth middleware", ...}
# exit: 0
```

Example failure invocation:

```
$ quest accept proj-a1                              # parent with non-terminal children
# stderr:
quest: conflict: proj-a1 has non-terminal children: proj-a1.1 (accepted), proj-a1.2 (open)
quest: exit 5 (conflict)
# stdout: (see spec — the conflict object is also emitted on stdout for structural conflicts)
{"error": "conflict", "task": "proj-a1", "non_terminal_children": [...]}
# exit: 5
```

### Debug Verbosity

With `--log-level debug`, slog surfaces internal milestones on stderr alongside the plain-text error line:

```
$ quest accept proj-a1 --log-level debug
# stderr:
level=DEBUG msg="quest command start" command=accept agent.role=planner dept.task.id=proj-a1
level=DEBUG msg="role gate passed" command=accept elevated=true
level=DEBUG msg="BEGIN IMMEDIATE acquired" tx_kind=accept_parent lock_wait_ms=1.2
level=INFO  msg="precondition failed" command=accept task.id=proj-a1 precondition=children_terminal blocked_by_ids=proj-a1.1,proj-a1.2
level=DEBUG msg="quest command complete" command=accept exit_code=5 duration_ms=4.1
quest: conflict: proj-a1 has non-terminal children: proj-a1.1 (accepted), proj-a1.2 (open)
quest: exit 5 (conflict)
```

The order is: slog records (as they happen), then the plain-text error explanation, then the exit tag. The plain-text line is always last on stderr, so callers that want just the summary can read the final line.

## Log Levels

Use these levels consistently. When in doubt, go one level quieter. This table aligns with the event inventory in OTEL.md §3.2 — if you are adding a new log call site, match an existing category there or add a row to it.

### `debug`

Internal state that is only useful when actively investigating a specific issue. Filtered out at both stderr (default `warn`) and the OTEL bridge (default `info`) by default.

Examples:

- Command handler entry and exit (lifecycle events).
- Resolved config values at startup.
- SQL statement trace (if ever enabled — currently excluded from telemetry per OTEL.md §4.5).

### `info`

Decisions and conflicts that an operator reviewing a session log would want to see.

Examples:

- Role-gate denials (`worker tried elevated command`).
- Preconditions failed (parent had non-terminal children on `accept`).
- Batch mode fell through with `--partial-ok` after validation errors.
- Schema migrations applied at startup (one record per invocation at most).

### `warn`

Something unexpected that the user can act on, but didn't stop the requested operation from making a correct response.

Examples:

- Batch validation produced errors (per-line WARN records alongside the stderr JSONL stream).
- Dependency cycle detected by `quest link` or `quest batch`.
- SQLite `busy_timeout` exceeded and the command returned exit 7.
- OTEL SDK internal error (collector unreachable, attribute rejected).
- Deprecated command, flag, or field was used.

### `error`

A bug. The current invocation cannot produce a correct response because something is broken. Reserved for exit-code-1 paths.

Examples:

- Panic recovered in a command handler.
- SQL error that was not mapped to a specific exit code.
- Migration failed with the database rolled back to the prior version.

### What does NOT get a log line

- Every successful SQL query. Query cost is captured on the command span and on `quest.store.op` events (OTEL.md §4.2). Per-query slog lines would drown the bridge.
- Stdout output. That is the command's result, not a log event.
- Routine control flow (entering a function, returning from a helper). `debug` is for state, not narration.

## Standard Field Names

Use these field names consistently in every slog call. Do not invent synonyms. This list extends OTEL.md §4.3 — the same names are used on spans, log records, and history entries wherever applicable.

| Field                  | Type    | Description                                                            |
| ---------------------- | ------- | ---------------------------------------------------------------------- |
| `command`              | string  | The quest subcommand (`accept`, `batch`, `create`, etc.).              |
| `agent.role`           | string  | Value of `AGENT_ROLE`, or `"unset"` if empty.                          |
| `dept.task.id`         | string  | Value of `AGENT_TASK`; omitted if empty.                               |
| `dept.session.id`      | string  | Value of `AGENT_SESSION`; omitted if empty.                            |
| `task.id`              | string  | The target task of this operation (may differ from `dept.task.id`).    |
| `duration_ms`          | float64 | Elapsed time in milliseconds.                                          |
| `exit_code`            | int     | Process exit code (0-7).                                               |
| `err`                  | string  | Error message, truncated to 256 characters. Never the raw `%v` of wrapped internal errors. |
| `tx_kind`              | string  | Structural transaction type (`accept_parent`, `move`, etc. — see OTEL.md). |
| `lock_wait_ms`         | float64 | Time spent waiting for the SQLite write lock.                          |
| `rows_affected`        | int     | Rows changed by a write or returned by a read.                         |
| `precondition`         | string  | Name of the precondition that failed (`children_terminal`, `parent_open`, etc.). |
| `blocked_by_ids`       | string  | Comma-joined list of task IDs that blocked this operation.             |
| `cycle.path`           | string  | Arrow-joined cycle path for dependency cycle errors.                   |
| `phase`                | string  | Batch validation phase (`parse`, `reference`, `graph`, `semantic`).    |
| `line`                 | int     | 1-based line number in a batch file.                                   |
| `code`                 | string  | Batch error code (see spec Batch error output).                        |
| `field`                | string  | The batch-row field that triggered a parse error.                      |
| `ref`                  | string  | Batch `ref` value when relevant.                                       |

### Naming Rules

- Use `snake_case` for field names. Dots are allowed as namespace separators (`agent.role`, `task.id`, `dept.task.id`) and follow the OTEL convention.
- Do not abbreviate (`duration_ms`, not `dur`). Do not nest structures inside a single field value — keep them flat.
- Scalar values only. Never pass a struct, a slice, or a map as a field value. If you need to attach a list, serialize it to a comma-joined string (`blocked_by_ids`) or split it into multiple records.
- Do not log free-form user content (task titles, descriptions, handoff text, debriefs, notes, reasons, metadata). Content capture is a span-event facility gated by `OTEL_GENAI_CAPTURE_CONTENT` — see OTEL.md §4.5. **Log attributes carry IDs, counts, enums, and error classes, never free-form content.**

## Boundary Logging

### Per-Invocation Bookends

Every command handler logs exactly once at entry and once at exit, at DEBUG level. These are bookends — not `info` — because the bookends exist for local development and tracing, not operator dashboards. The OTEL command span carries the same information at a higher signal-to-noise ratio.

```
level=DEBUG msg="quest command start" command=create agent.role=planner dept.task.id=proj-a1
level=DEBUG msg="quest command complete" command=create exit_code=0 duration_ms=12.4
```

### Per-Transaction Boundaries

Structural transactions (those that use `BEGIN IMMEDIATE`) get a lifecycle pair at DEBUG:

```
level=DEBUG msg="BEGIN IMMEDIATE acquired" tx_kind=accept_parent lock_wait_ms=1.2
level=DEBUG msg="tx committed"             tx_kind=accept_parent rows_affected=2
```

On rollback:

```
level=DEBUG msg="tx rolled back" tx_kind=accept_parent outcome=rolled_back_precondition
```

Do not log inside the transaction for every SQL statement. Per-statement cost is captured as span events per OTEL.md §4.2.

### Decisions and Conflicts

Decisions that change control flow get an INFO record with the cause, matching the OTEL log inventory:

```
level=INFO  msg="role gate denied" command=move agent.role=coder required=elevated
level=INFO  msg="precondition failed" command=accept task.id=proj-a1 precondition=children_terminal blocked_by_ids=proj-a1.1,proj-a1.2
```

## Error Handling

### Wrapping

Use `fmt.Errorf` with `%w` so the chain of causation is preserved:

```go
// CORRECT
if err := h.store.Accept(ctx, taskID, sessionID); err != nil {
    return protocol.ErrorResponse(ErrConflict, fmt.Errorf("accept task %s: %w", taskID, err))
}

// WRONG — loses the original error
if err != nil {
    return protocol.ErrorResponse(ErrInternal, fmt.Errorf("accept failed"))
}
```

Log the full wrapped error at the boundary (where it becomes a non-zero exit) via slog, then sanitize before emitting to the user-facing channel:

```go
slog.ErrorContext(ctx, "internal error",
    "command", cmd,
    "task.id", taskID,
    "err", truncate(err.Error(), 256),
)
fmt.Fprintf(os.Stderr, "quest: %s: %s\n", errorClassFor(err), userMessageFor(err))
```

### Sanitization

The user-facing stderr message MUST:

- Name the error class (from the exit-code mapping).
- State what the user can do next, if anything.
- NOT include SQL statements, file offsets, internal type names, or stack traces.

The full wrapped error goes to slog (which flows to the OTEL log bridge) and to the span's recorded error. Internal detail is not hidden — it is in the right channel.

Wrong:

```
quest: internal_error: sql: no rows in result set at /home/you/go/pkg/mod/modernc.org/sqlite@v1.28.0/lib/...
```

Right:

```
quest: not_found: task proj-a1 does not exist
quest: exit 3 (not_found)
```

The slog record carries the underlying wrapped error; the user-facing line carries the class and the fix.

### Exit Codes

Exit codes 1-7 are a stable contract (see `quest-spec.md` Output & Error Conventions and `STANDARDS.md` Part 2). They map 1:1 to `quest.error.class` in `OTEL.md §4.4`. A single helper owns the mapping:

```go
// internal/errors/exit.go
func ExitCode(err error) int { /* ... */ }
func Class(err error) string { /* general_failure, usage_error, ... */ }
```

Every call site that terminates with a non-zero exit uses this helper. Do not hardcode exit numbers in individual commands.

## Anti-Patterns

These are patterns that agents and developers MUST NOT use. If you see these in the codebase, fix them.

### 1. `fmt.Println` / `fmt.Fprintln(os.Stderr)` for diagnostics

```go
// WRONG — bypasses slog, bypasses the OTEL bridge, cannot be level-gated
fmt.Fprintln(os.Stderr, "accepting task...")

// CORRECT
slog.DebugContext(ctx, "handling accept", "command", "accept", "task.id", taskID)
```

Plain-text stderr output is reserved for the final `quest: <class>: <message>` error line and the batch stderr JSONL stream. Everything else is slog.

### 2. `fmt.Println` for command results

```go
// WRONG — bypasses --format, skips json/text branching
fmt.Println(taskJSON)

// CORRECT — every command output path goes through internal/output
output.Emit(cfg.Format, result)
```

### 3. Unstructured logging

```go
// WRONG — values embedded in the message string can't be filtered or queried
slog.Info("accepted task " + taskID + " in " + strconv.Itoa(ms) + "ms")

// CORRECT
slog.InfoContext(ctx, "task accepted", "task.id", taskID, "duration_ms", ms)
```

### 4. Logging without context

```go
// WRONG — no trace_id or span_id will be attached
slog.Info("processing batch")

// CORRECT
slog.InfoContext(ctx, "processing batch", "lines", n)
```

### 5. Per-step logs at info level

```go
// WRONG — drowns real INFO events
slog.InfoContext(ctx, "opened database")
slog.InfoContext(ctx, "began transaction")
slog.InfoContext(ctx, "inserted row")

// CORRECT — boundary only; per-statement cost is already captured as span events
slog.DebugContext(ctx, "tx committed", "tx_kind", "create_child", "rows_affected", 1)
```

### 6. Inconsistent field names

```go
// WRONG — same concept, different names
slog.InfoContext(ctx, "started", "taskID", id)
slog.InfoContext(ctx, "finished", "task_id", id)

// CORRECT — use the standard field from the table
slog.InfoContext(ctx, "started", "task.id", id)
slog.InfoContext(ctx, "finished", "task.id", id)
```

### 7. Swallowing errors

```go
// WRONG
result, _ := h.store.Load(ctx, id)

// CORRECT
result, err := h.store.Load(ctx, id)
if err != nil {
    return protocol.ErrorResponse(ErrInternal, fmt.Errorf("load %s: %w", id, err))
}
```

### 8. Logging user content

```go
// WRONG — debrief and handoff are content, not attributes
slog.InfoContext(ctx, "task completed", "debrief", task.Debrief, "handoff", task.Handoff)

// CORRECT — content is span-event-only, gated by OTEL_GENAI_CAPTURE_CONTENT (OTEL.md §4.5).
slog.InfoContext(ctx, "task completed", "task.id", id, "debrief_len", len(task.Debrief))
```

### 9. Raw exit codes

```go
// WRONG — exit codes scattered across handlers drift out of sync with the spec
os.Exit(5)

// CORRECT — single source of truth
os.Exit(errors.ExitCode(err))
```

## Agent Instructions Summary

If you are a coding agent working on this codebase, here is your checklist:

1. **Import `internal/logging`** for logger setup. Do not construct `slog.Logger` directly.
2. **Import `internal/errors` (or whatever it is called)** for error classification and exit-code mapping. Never hardcode an exit number.
3. **Always use `slog.InfoContext` / `ErrorContext` / etc.** — never the non-Context variants inside handlers. Trace and span IDs flow through context.
4. **Do not invent a correlation ID.** Use `AGENT_TASK`, `AGENT_SESSION`, and the trace context already in the environment.
5. **Stdout is for data.** Everything else goes to stderr.
6. **Log at the boundary.** Command entry, command exit, transaction start, transaction end, decisions that change control flow. Do not narrate per-SQL steps at INFO.
7. **Use the standard field names** from the table above. No abbreviations, no synonyms, no structured values.
8. **Never log free-form user content.** Titles, descriptions, debriefs, notes, and handoff text are content. Content capture is span-event-only (OTEL.md §4.5).
9. **Wrap errors** with `%w` and include the operation and task ID in the context string.
10. **Sanitize user-facing error messages.** Internal detail goes to slog; the `quest: <class>: <message>` line names the class and the fix.
11. **Never log secrets, tokens, or credentials** at any level. Quest does not accept secrets as input, so this is mostly prevention — if a future feature introduces any, redact before logging.
12. **When adding a new log call site,** match an existing category in `OTEL.md §3.2` or propose a new one. Do not freelance events.
