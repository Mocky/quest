# Testing Standard

This document defines the testing conventions for quest. It serves the same dual purpose as `OBSERVABILITY.md`: guiding human developers and instructing coding agents that work on this codebase.

All tests — whether written by a human or generated by an agent — must follow these conventions. If you are a coding agent, treat every MUST/MUST NOT in this document as a hard constraint.

The behavioral contract being tested is `quest-spec.md`. When a test and the spec disagree, the spec wins and the test gets updated.

---

## Philosophy

Go's testing culture is deliberately minimal. This project follows that culture.

1. **Use the standard library.** `testing`, `os/exec`, `encoding/json`, `database/sql`, `io`, `context`. Do not introduce `testify`, `gomock`, `gocheck`, or any third-party test framework. If you are a coding agent and you are tempted to write `require.Equal`, stop. Use an `if` statement.
2. **Table-driven tests are the default pattern.** If a function can be tested with multiple inputs and expected outputs, use a table. No exceptions.
3. **Subtests provide structure.** Use `t.Run(name, func(t *testing.T) { ... })` to label each case. This gives you filterable test names and per-case failure isolation.
4. **Test behavior, not implementation.** Assert on what a command returns (exit code, stdout JSON, database state) and what invariants it preserves, not on which internal function was called. This keeps tests stable across refactors.
5. **Tests are documentation.** A reader should be able to understand how a command behaves by reading its test cases. Name cases clearly: `"accept on parent with non-terminal child returns exit 5"`, not `"test case 3"`.
6. **The spec is load-bearing.** Many tests are contract tests derived from the spec: field presence, exit-code mapping, idempotency, error output shape. Treat spec text as a specification of test assertions — if the spec says "all fields are always present in JSON output," every command's JSON output has a test that enforces it.

## Test Layers

Quest has five test layers. Every significant package should have coverage at the appropriate layer(s).

### Layer 1 — Unit Tests

Test individual functions and methods in isolation. No database, no filesystem outside `t.TempDir()`, no subprocess invocation.

**What belongs here:**

- Pure validation functions (ID prefix, task-ID parsing, batch-row parsing per line).
- Cycle detection over an in-memory dependency graph.
- Depth computation and enforcement (ID "proj-a1.1.1" → depth 3).
- Config loading and validation (with `t.Setenv` for env-var parsing tests).
- Exit-code mapping from wrapped errors.
- Log-level and format parsing.
- Role-gate resolution given (`AGENT_ROLE`, `elevated_roles`).
- Text-mode column truncation and rendering.
- Output-format selection and JSON marshaling helpers.

**File naming:** `*_test.go` in the same package as the code being tested.

**Example scope:** Given a validator, table-test that `proj-a1.1.1` parses to depth 3, that `proj` is an invalid task ID (missing short ID), and that `ref` is a rejected prefix.

### Layer 2 — Contract Tests

Test the agent-facing surface: JSON output shape, exit-code mapping, history entry shape, export file layout, idempotency guarantees. These are the tripwires for breaking changes.

**What belongs here:**

- **JSON output field presence.** For every command, assert that the output JSON has every field listed in the spec, even when the value is `null` or `[]`.
- **Exit-code stability.** A table test pinning each exit code (1-7) to its class string (`general_failure`, `usage_error`, ..., `transient_failure`). If someone renumbers, this test fails before the code ships.
- **Idempotency guarantees.** For each command in the spec's idempotency table, assert that the documented retry behavior holds (duplicate PR ignored, duplicate tag is a no-op, etc.).
- **History entry shape.** Assert that `action`, `role`, `session`, and action-specific fields (`reason`, `fields`, `content`, `target`, `link_type`, `old_id`, `new_id`) appear when and only when the spec says.
- **Batch stderr JSONL format.** For every batch error code (`malformed_json`, `missing_field`, `duplicate_ref`, etc.), assert the exact field set on the stderr line.
- **Export file layout.** Assert that `quest export` produces `tasks/{id}.json`, `debriefs/{id}.md`, and `history.jsonl` with the documented schemas.
- **Role-gate denials.** Assert that each elevated command returns exit 6 with the documented error shape when a worker role invokes it.

**File naming:** `contract_test.go` in the package that owns the surface (e.g., `internal/cli/contract_test.go`, `internal/batch/contract_test.go`).

**Why this layer matters:** Agents parse this output. If serialization breaks, every dashboard, every retrospective query, and every retry heuristic breaks silently. Contract tests are cheap; breakage is expensive.

### Layer 3 — Database Integration Tests

Test handlers against a real SQLite database. The database is created in `t.TempDir()`, seeded from fixtures or from direct SQL, torn down automatically.

**What belongs here:**

- Every command handler against a real `quest.db`.
- Structural transaction correctness (`BEGIN IMMEDIATE`): no interleaving, no TOCTOU, atomic rollback on precondition failure.
- Dependency graph operations over real tables: cycle detection on `link`, depth check on `create --parent` and `move`, subgraph rename on `move`.
- Migration correctness: fixture database at `schema_version = N` opens cleanly with a binary supporting `N+1`.
- `PRAGMA busy_timeout = 5000` behavior: concurrent writers actually wait up to 5 seconds and then exit 7.
- WAL-mode readers: a long-running `quest list` does not block a concurrent `quest create`.

**File naming:** `store_test.go` or `handler_test.go` in the relevant `internal/` package. Integration tests that need concurrency or multiple simultaneous DB connections MUST use the `//go:build integration` tag so they do not run during fast `go test ./...` cycles.

**Example scope:** Given two goroutines both calling `quest accept` on the same open task, exactly one succeeds, the other receives exit 5 (conflict). The successful transaction is atomic: `owner_session` and `started_at` are set together or not at all.

### Layer 4 — CLI Output Contract Tests

Test what agents actually see: stdout JSON, stderr diagnostics, exit codes. These tests build and invoke the real binary.

**What belongs here:**

- Success output shape: valid JSON on stdout, no envelope wrapper, no trailing whitespace games.
- Error output shape: `quest: <class>: <message>` on stderr as the final line, plus the structured JSON body where the spec requires it (e.g., conflict on `accept` includes the non-terminal children on stdout).
- Exit codes: 0 on success; 1-7 for failures, mapped per the spec.
- Batch stderr JSONL: one object per error, with the documented fields.
- `--text` rendering: tables for `list` and `graph`, indented tree for `graph`, column truncation when output is piped vs. TTY-sized when a TTY.
- `--log-level` raises or lowers slog verbosity as expected.
- Environment variable resolution: `AGENT_TASK` as default task ID, `AGENT_ROLE` for gating.

**File naming:** `cli_test.go` in a package that can build and invoke the quest binary. Use a `TestMain` that runs `go build` once into a `t.TempDir()` and points subsequent `os/exec.Command` calls at it. Tag with `//go:build integration` if the build step is slow enough to matter for the fast cycle.

**Why this layer matters:** The JSON output contract layer catches shape regressions, but a CLI can still break in subtle ways — wrong stream for an error, exit code not propagated from a handler, `--text` ignored in some code path. Layer 4 catches those.

### Layer 5 — Concurrency & Load Tests

Test quest's behavior under agent-swarm conditions: many concurrent writers, sustained write load, busy-timeout boundary.

**What belongs here:**

- Agent-swarm simulation: N goroutines each running a mix of `create`, `accept`, `update`, `complete`. Assert no data races (run with `-race`), no lost updates, no phantom rows.
- Busy-timeout boundary: a long-running write holds the lock; a parallel writer waits up to 5 seconds, then exits 7. Measure the distribution of wait times matches the spec.
- Structural transaction serialization: concurrent `quest accept` on the same parent (all with non-terminal siblings reaching terminal at random times) produces correct, non-interleaved outcomes.
- Dependency-graph stress: a batch of 500 tasks with dense `blocked-by` edges validates in reasonable time; no cycle check false positives or negatives.

**File naming:** `concurrency_test.go` in `internal/integration/` or alongside the storage package. Always `//go:build integration`.

**Run expectations:** These tests are slow and resource-hungry; they run in CI but not on every `go test ./...`. Run explicitly with `go test -tags integration -race ./...`.

## Patterns and Conventions

### Table-Driven Tests

This is the primary test pattern. Every test that checks multiple cases MUST use this structure.

```go
func TestValidatePrefix(t *testing.T) {
    tests := []struct {
        name    string
        prefix  string
        wantErr string // empty for success
    }{
        {"simple lowercase", "proj", ""},
        {"letters and digits", "p2", ""},
        {"min length", "ab", ""},
        {"max length", "abcdefgh", ""},
        {"too short", "p", "must be 2-8 characters"},
        {"too long", "abcdefghi", "must be 2-8 characters"},
        {"starts with digit", "1proj", "must start with a letter"},
        {"contains hyphen", "my-proj", "lowercase letters and digits only"},
        {"contains dot", "my.proj", "lowercase letters and digits only"},
        {"uppercase", "Proj", "lowercase letters and digits only"},
        {"reserved ref", "ref", "reserved prefix"},
    }

    for _, tt := range tests {
        t.Run(tt.name, func(t *testing.T) {
            err := ValidatePrefix(tt.prefix)
            if tt.wantErr == "" {
                if err != nil {
                    t.Fatalf("ValidatePrefix(%q) unexpected error: %v", tt.prefix, err)
                }
                return
            }
            if err == nil {
                t.Fatalf("ValidatePrefix(%q) error = nil, want error containing %q", tt.prefix, tt.wantErr)
            }
            if !strings.Contains(err.Error(), tt.wantErr) {
                t.Errorf("ValidatePrefix(%q) error = %q, want substring %q",
                    tt.prefix, err.Error(), tt.wantErr)
            }
        })
    }
}
```

### Assertions Without Testify

Go tests use `if` statements and `t.Errorf` / `t.Fatalf`. Idiomatic equivalents:

```go
// Equality
if got != want {
    t.Errorf("FuncName() = %v, want %v", got, want)
}

// Deep equality (for structs, slices, maps)
if !reflect.DeepEqual(got, want) {
    t.Errorf("FuncName() = %+v, want %+v", got, want)
}

// Error expected
if err == nil {
    t.Fatal("FuncName() error = nil, want error")
}

// No error expected
if err != nil {
    t.Fatalf("FuncName() unexpected error: %v", err)
}

// Contains (for error messages, output strings)
if !strings.Contains(got, want) {
    t.Errorf("output = %q, want substring %q", got, want)
}

// JSON field presence
var parsed map[string]any
if err := json.Unmarshal(stdout, &parsed); err != nil {
    t.Fatalf("stdout is not valid JSON: %v", err)
}
if _, ok := parsed["handoff_written_at"]; !ok {
    t.Error("required field \"handoff_written_at\" missing from output")
}
```

**Use `t.Errorf` for non-fatal assertions** (test continues, reports all failures). **Use `t.Fatalf` for fatal assertions** (test stops because continuing would panic or be meaningless — typically after checking `err != nil` before using a result).

### Test Helpers

Shared helpers go in `internal/testutil/`. Mark every helper with `t.Helper()` so failure messages report the caller's line, not the helper's.

```go
// NewWorkspace creates a temp directory with a valid .quest/config.toml.
// Returns the absolute workspace path.
func NewWorkspace(t *testing.T, prefix string) string {
    t.Helper()
    dir := t.TempDir()
    if err := os.Mkdir(filepath.Join(dir, ".quest"), 0o755); err != nil {
        t.Fatalf("mkdir .quest: %v", err)
    }
    body := fmt.Sprintf("id_prefix = %q\nelevated_roles = [\"planner\"]\n", prefix)
    if err := os.WriteFile(filepath.Join(dir, ".quest/config.toml"), []byte(body), 0o644); err != nil {
        t.Fatalf("write config.toml: %v", err)
    }
    return dir
}

// AssertExitCode runs a quest command and asserts the process exit code.
func AssertExitCode(t *testing.T, cmd *exec.Cmd, want int) (stdout, stderr []byte) {
    t.Helper()
    var outBuf, errBuf bytes.Buffer
    cmd.Stdout = &outBuf
    cmd.Stderr = &errBuf
    err := cmd.Run()
    got := 0
    if ee, ok := err.(*exec.ExitError); ok {
        got = ee.ExitCode()
    } else if err != nil {
        t.Fatalf("command %v failed to run: %v", cmd.Args, err)
    }
    if got != want {
        t.Errorf("exit code = %d, want %d\nstdout: %s\nstderr: %s",
            got, want, outBuf.String(), errBuf.String())
    }
    return outBuf.Bytes(), errBuf.Bytes()
}

// AssertErrorClass asserts that stderr ends with `quest: <class>: ...` then `quest: exit N (<class>)`.
func AssertErrorClass(t *testing.T, stderr []byte, class string) {
    t.Helper()
    // ...
}
```

### Store Fixtures and Seed Helpers

Tests that need a populated database get a helper that opens a fresh SQLite file and applies a seed:

```go
// NewStore opens an empty quest.db at migration head inside t.TempDir().
// Returns the store.Store interface (not a concrete type) — handlers and
// tests both depend on the interface so InstrumentedStore can decorate it.
func NewStore(t *testing.T) store.Store {
    t.Helper()
    path := filepath.Join(t.TempDir(), "quest.db")
    s, err := store.Open(path)
    if err != nil {
        t.Fatalf("store.Open: %v", err)
    }
    if err := store.Migrate(context.Background(), s); err != nil {
        t.Fatalf("store.Migrate: %v", err)
    }
    t.Cleanup(func() { s.Close() })
    return s
}

// SeedTask inserts a task directly (bypassing CLI/handler) for setup in handler tests.
func SeedTask(t *testing.T, s store.Store, task store.Task) {
    t.Helper()
    // ...
}
```

Tests that need a CLI binary use a `TestMain` to build once per package:

```go
var questBin string

func TestMain(m *testing.M) {
    tmp, err := os.MkdirTemp("", "quest-bin-")
    if err != nil {
        fmt.Fprintln(os.Stderr, "mktemp:", err)
        os.Exit(1)
    }
    defer os.RemoveAll(tmp)

    questBin = filepath.Join(tmp, "quest")
    cmd := exec.Command("go", "build", "-o", questBin, "./cmd/quest")
    cmd.Stderr = os.Stderr
    if err := cmd.Run(); err != nil {
        fmt.Fprintln(os.Stderr, "build:", err)
        os.Exit(1)
    }
    os.Exit(m.Run())
}
```

Call the binary as `exec.Command(questBin, "show", "proj-01")` with env vars set per-case.

### Test Fixtures

For tests that need predefined data (batch JSONL files, expected JSON responses, history snapshots), place fixtures in a `testdata/` directory within the package. Go's tooling ignores `testdata/` directories during builds.

```
internal/batch/
├── batch.go
├── batch_test.go
└── testdata/
    ├── valid_simple.jsonl
    ├── valid_with_refs.jsonl
    ├── cycle_error.jsonl
    ├── depth_exceeded.jsonl
    └── missing_title.jsonl
```

Load fixtures with `os.ReadFile`:

```go
func loadFixture(t *testing.T, name string) []byte {
    t.Helper()
    data, err := os.ReadFile(filepath.Join("testdata", name))
    if err != nil {
        t.Fatalf("failed to load fixture %s: %v", name, err)
    }
    return data
}
```

Fixture databases for migration tests live in `internal/store/testdata/migrations/` with filenames like `schema_1.db`, `schema_2.db`, etc. Keep them small; generate them with a documented script (checked in alongside).

### Concurrency Tests

Concurrency tests MUST use `-race` (enforced in CI). Use `sync.WaitGroup` to coordinate, channels for results, and avoid `time.Sleep` as a coordination mechanism.

```go
//go:build integration

func TestConcurrentAcceptLeavesOnlyOneWinner(t *testing.T) {
    s := testutil.NewStore(t)
    cfg := testutil.NewConfig(t)
    testutil.SeedTask(t, s, store.Task{ID: "tst-01", Status: "open"})

    var wg sync.WaitGroup
    results := make(chan error, 10)
    for i := 0; i < 10; i++ {
        wg.Add(1)
        go func(i int) {
            defer wg.Done()
            // testutil.CallAccept invokes the accept handler directly. quest's Store
            // interface does not expose a coarse Accept method (handler-owns-SQL model);
            // each goroutine drives a fresh handler invocation with its own session ID.
            results <- testutil.CallAccept(t, s, cfg, "tst-01", fmt.Sprintf("sess-%d", i))
        }(i)
    }
    wg.Wait()
    close(results)

    var wins, conflicts int
    for err := range results {
        switch {
        case err == nil:
            wins++
        case errors.Is(err, errors.ErrConflict):
            conflicts++
        default:
            t.Errorf("unexpected error: %v", err)
        }
    }
    if wins != 1 {
        t.Errorf("winners = %d, want 1", wins)
    }
    if conflicts != 9 {
        t.Errorf("conflicts = %d, want 9", conflicts)
    }
}
```

Notes on the helpers used above:

- `testutil.CallAccept(t, s, cfg, taskID, sessionID)` invokes the accept handler with a freshly-constructed config carrying `AGENT_SESSION=sessionID`. quest's `Store` interface deliberately does **not** expose write methods like `Accept`/`UpdateFields` — write logic lives in handlers, not the store (handler-owns-SQL model). The helper is the test-time analogue of dispatching the command end-to-end without going through `os/exec`.
- Sentinel errors live in `internal/errors/` (`errors.ErrConflict`, `errors.ErrNotFound`, etc.), **not** `internal/store/`. The store maps `SQLITE_BUSY` and constraint violations to the same `errors.Err*` sentinels at the boundary.
- See the implementation plan's Task 3.3 (Store interface) and Task 2.2 (sentinel errors) for the design rationale.

### Testing Env Var Resolution

Config tests that exercise `Load()` use `t.Setenv` so the test harness restores the prior value:

```go
func TestLoadReadsAgentRole(t *testing.T) {
    t.Setenv("AGENT_ROLE", "planner")
    // ...
}
```

Handler tests do NOT use `t.Setenv`. They construct a `config.Config` directly with the values the case needs. This keeps the handler test surface independent of process state and avoids flakiness when tests run in parallel.

## Test Naming

### File Names

| File                    | Contents                                                       |
| ----------------------- | -------------------------------------------------------------- |
| `foo_test.go`           | Unit tests for `foo.go`                                        |
| `contract_test.go`      | JSON/exit-code/idempotency contract tests                      |
| `store_test.go`         | Database-backed integration tests                              |
| `cli_test.go`           | End-to-end CLI output tests (builds and invokes the binary)    |
| `concurrency_test.go`   | Agent-swarm and lock-contention tests (integration-tagged)     |
| `migration_test.go`     | Schema migration tests against fixture databases               |

### Test Function Names

Use `Test<Function>` for unit tests and `Test<Scenario>` for contract/integration/cli tests:

```go
// Unit tests — named after the function
func TestValidatePrefix(t *testing.T) { ... }
func TestDetectCycle(t *testing.T) { ... }
func TestExitCodeForError(t *testing.T) { ... }

// Contract tests — named after the scenario
func TestShowJSONHasRequiredFields(t *testing.T) { ... }
func TestExitCodeStability(t *testing.T) { ... }
func TestBatchErrorStderrShape(t *testing.T) { ... }

// Integration tests — named after the scenario
func TestAcceptParentWithNonTerminalChildFails(t *testing.T) { ... }
func TestMoveRewritesSubgraphIDs(t *testing.T) { ... }
func TestBusyTimeoutTransientFailure(t *testing.T) { ... }
```

### Subtest Names

Use lowercase, descriptive phrases. These become part of the `-run` filter, so make them greppable:

```go
t.Run("open leaf transitions to accepted", ...)
t.Run("parent with non-terminal children returns exit 5", ...)
t.Run("already-cancelled task returns exit 0", ...)
t.Run("batch with cycle reports graph-phase error", ...)
```

## What to Test in Each Package

### `internal/config/`

- Prefix validation: table-test every rule from the spec's Prefix validation section (length, characters, reserved values).
- Config file parsing: valid `.quest/config.toml`, missing file, malformed TOML, unknown fields (warn but continue).
- Env var resolution: `AGENT_ROLE`, `AGENT_TASK`, `AGENT_SESSION` empty vs. set.
- Flag overrides: `--log-level` overrides `QUEST_LOG_LEVEL` overrides default.
- `Validate()`: collects multiple errors into a single message.

### `internal/logging/`

- Level parsing: `"debug"` → `LevelDebug`, `"DEBUG"` → `LevelDebug`, `"garbage"` → error or default, empty → default.
- Format parsing: same pattern.
- Context round-trip: writing and reading task/session IDs on context.
- Fan-out handler: both children receive records; stderr-level and OTEL-level gate independently.

### `internal/store/` (or wherever SQLite access lives)

- Every write operation: happy path plus every precondition failure listed in the spec.
- Structural transactions: `BEGIN IMMEDIATE` acquires the write lock; failed precondition rolls back cleanly; successful precondition commits atomically.
- Cycle detection on `link --blocked-by`.
- Depth enforcement on `create --parent` and `move`.
- Subgraph rename on `move`: task IDs and all incoming/outgoing dependency references updated atomically.
- Migration: fixture DB at schema N opens with binary supporting N+1; fixture DB at schema N+1 refuses to open with binary supporting N.
- Busy timeout: second writer waits up to 5 seconds, then returns the transient-failure error.

### Handler Unit Tests

- Each command handler gets a test function with a table covering: valid input, each validation failure case, each precondition failure, each role-gate path, and edge cases (empty strings, unicode, very long input within spec limits).
- Always assert on: exit code (via error class), stdout JSON shape (via contract helpers), database state after the command, history entries appended.
- Do not test: log output content, internal function call counts, performance (unless there is a specific latency budget in the spec).

### `internal/batch/`

- Parse phase: malformed JSON, missing required fields, empty file, blank-line skipping with correct line numbers for subsequent errors.
- Reference phase: duplicate ref, unresolved ref, unknown external task ID.
- Graph phase: cycles (inside the batch, across the batch and existing graph), depth exceeded.
- Semantic phase: every target status constraint, every source type constraint (the full table from the spec's Dependency validation section).
- Cross-phase isolation: a line that fails parse is excluded from reference/graph/semantic evaluation.
- Atomic vs. `--partial-ok`: no tasks created on any error vs. partial creation with dependency-connected rollback.

### CLI Output Contract Tests

- Success output shape: invoke the CLI against a fixture workspace, capture stdout, parse as JSON, verify every spec-mandated field is present.
- Error output shape: invoke with bad input, capture both streams, verify stderr ends with `quest: <class>: <message>` then `quest: exit N (<class>)`, and the exit code matches.
- Batch stderr JSONL: every batch error code produces a line with the documented fields.
- `--text`: table rendering matches the spec's column defaults and truncation rules; `graph` indentation matches.
- stderr request context (in DEBUG mode): `dept.task.id` and `dept.session.id` appear on slog records when the env vars are set, are absent otherwise.

### `internal/export/`

- Layout: `tasks/{id}.json`, `debriefs/{id}.md` (only for tasks with debriefs), `history.jsonl` (chronological across all tasks).
- Task JSON is identical to `quest show --history` output.
- History JSONL preserves the order of history entries by timestamp.
- Idempotency: re-running `quest export` overwrites the output directory without accumulating stale files.

## Running Tests

### Fast feedback (unit + contract tests)

```bash
make test
# or: go test ./...
```

Runs without build tags. Fast, no concurrency stress, no subprocess builds.

### Full suite (including integration and CLI tests)

```bash
make test-all
# or: go test -tags integration ./...
```

### Single test or subtest

```bash
go test ./internal/store/ -run TestAcceptParentWithNonTerminalChildFails
go test ./internal/batch/ -run "TestBatchValidation/cycle_in_blocked_by"
```

### With race detection (ALWAYS run in CI)

```bash
go test -race -tags integration ./...
```

Race detection is not optional for quest. Structural transactions and concurrency correctness are core invariants — a race regression is a correctness regression.

### With coverage

```bash
go test -coverprofile=coverage.out ./...
go tool cover -html=coverage.out -o coverage.html
```

## CI Expectations

Tests in CI should:

1. Run with `-race` to catch data races.
2. Run with `-tags integration` to include integration and CLI tests.
3. Run with `-count=1` to disable test caching.
4. Run the contract test suite unconditionally — even on branches that don't change handler code — so that refactors in adjacent packages can't silently break the agent-facing surface.
5. Produce coverage reports. There is no hard coverage threshold — coverage is a tool for finding untested code, not a metric to game.
6. Fail the build on any test failure. No skipped or ignored tests in CI without a tracking task (logged via `quest create` against the project itself, naturally).

```bash
# Recommended CI command
go test -race -count=1 -tags integration -coverprofile=coverage.out ./...
```

## Anti-Patterns

These are patterns that agents and developers MUST NOT use. If you see these in the codebase, fix them.

### 1. Third-party assertion libraries

```go
// WRONG — do not use testify
import "github.com/stretchr/testify/assert"
assert.Equal(t, expected, actual)

// CORRECT — standard library
if got != want {
    t.Errorf("got %v, want %v", got, want)
}
```

### 2. Non-table-driven tests with repeated logic

```go
// WRONG — copy-paste test cases
func TestValidatePrefix(t *testing.T) {
    if err := ValidatePrefix("proj"); err != nil { t.Error(err) }
    if err := ValidatePrefix("p1"); err != nil { t.Error(err) }
}

// CORRECT — table-driven, with named subtests
```

### 3. Unnamed or generic subtest names

```go
// WRONG
t.Run("case 1", ...)
t.Run("test", ...)

// CORRECT — filterable, descriptive
t.Run("empty prefix returns error", ...)
t.Run("accept on cancelled task returns exit 5", ...)
```

### 4. Tests that share global state

```go
// WRONG — TestB depends on TestA running first
var sharedStore *store.Store

func TestA(t *testing.T) { sharedStore = NewStore() }
func TestB(t *testing.T) { useStore(sharedStore) }

// CORRECT — each test owns its own store
func TestB(t *testing.T) {
    s := NewStore(t)
    useStore(s)
}
```

### 5. `os.Setenv` in handler tests

```go
// WRONG — leaks into other parallel tests
func TestAccept(t *testing.T) {
    os.Setenv("AGENT_ROLE", "planner")
    // ...
}

// CORRECT — pass values through Config
func TestAccept(t *testing.T) {
    cfg := config.Config{Agent: config.AgentConfig{Role: "planner"}}
    // ...
}

// OK — for testing Load() itself, use t.Setenv which auto-restores
func TestLoadReadsAgentRole(t *testing.T) {
    t.Setenv("AGENT_ROLE", "planner")
    // ...
}
```

### 6. `time.Sleep` as a synchronization primitive

```go
// WRONG — flaky under load, slow under no load
go startWriter(ctx)
time.Sleep(100 * time.Millisecond)
// now try concurrent writer...

// CORRECT — synchronize on a channel or a `sync.WaitGroup`
started := make(chan struct{})
go func() { close(started); startWriter(ctx) }()
<-started
// now try concurrent writer...
```

### 7. Missing `t.Helper()` on helpers

```go
// WRONG — failure messages point to the helper line, not the caller
func assertExit(t *testing.T, got, want int) {
    if got != want { t.Fatalf("exit = %d, want %d", got, want) }
}

// CORRECT
func assertExit(t *testing.T, got, want int) {
    t.Helper()
    if got != want { t.Fatalf("exit = %d, want %d", got, want) }
}
```

### 8. Testing implementation instead of behavior

```go
// WRONG — tests that the handler called a specific internal method
if !store.insertTaskWasCalled {
    t.Error("expected insertTask to be called")
}

// CORRECT — tests the observable outcome
row, err := store.Load(ctx, "proj-01")
if err != nil || row.Status != "open" {
    t.Errorf("after create, task row = %+v, want open", row)
}
```

### 9. Ignoring errors in test setup

```go
// WRONG — hides setup failures
file, _ := os.Create(filepath.Join(dir, "batch.jsonl"))
file.Write(data)

// CORRECT — setup failures are test failures, not silent bugs
file, err := os.Create(filepath.Join(dir, "batch.jsonl"))
if err != nil {
    t.Fatalf("create fixture: %v", err)
}
t.Cleanup(func() { file.Close() })
if _, err := file.Write(data); err != nil {
    t.Fatalf("write fixture: %v", err)
}
```

### 10. Integration tests without build tags

```go
// WRONG — runs on every go test ./..., slow, may fail without binary
func TestFullCLIRoundTrip(t *testing.T) {
    // builds binary, runs it...
}

// CORRECT — only runs when explicitly requested
//go:build integration

func TestFullCLIRoundTrip(t *testing.T) {
    // ...
}
```

### 11. Asserting on stderr message text

```go
// WRONG — stderr messages are for humans; they will change
if !strings.Contains(stderr, "task proj-01 does not exist") {
    t.Error("wrong error message")
}

// CORRECT — assert on the error class and exit code (both are contracts)
assertExitCode(t, cmd, 3)        // 3 = not_found
assertErrorClass(t, stderr, "not_found")
```

The only text assertion that is stable is the final-line tag — `quest: exit N (<class>)`. Everything else on stderr is slog records or a one-line human explanation that may be reworded freely.

### 12. Asserting on log record content

```go
// WRONG — coupling tests to slog output couples tests to logging decisions
if !strings.Contains(stderr, `level=DEBUG msg="tx committed"`) {
    t.Error("expected transaction log")
}

// CORRECT — assert on the observable outcome (database state, exit code, stdout)
```

## Agent Instructions Summary

If you are a coding agent working on this codebase, here is your checklist:

1. **Never import a test framework.** Use `testing`, `if` statements, and `t.Errorf` / `t.Fatalf`. No `testify`, no `gomock`.
2. **Always use table-driven tests** when there are multiple input/output cases. Define a `tests` slice of structs, iterate with `for _, tt := range tests`, and wrap each case in `t.Run(tt.name, ...)`.
3. **Name subtests descriptively.** Filterable, greppable phrases: `"accept on parent with non-terminal child returns exit 5"`.
4. **Call `t.Helper()`** in every test helper function, on the first line.
5. **Put test helpers** in `internal/testutil/`. Do not duplicate helper logic across test files.
6. **Put test fixtures** in `testdata/` directories within the relevant package.
7. **Tag integration, CLI, and concurrency tests** with `//go:build integration` at the top of the file.
8. **Test the behavior, not the implementation.** Assert on exit codes, stdout JSON shape, database state, history entries — never on log lines or internal call counts.
9. **Use `t.Fatalf` when continuing makes no sense** (e.g., after an unexpected error). Use `t.Errorf` otherwise so all failures are reported.
10. **Always check errors** in setup, teardown, and assertions. Never use `_` for an error return in test code.
11. **Test both success and failure paths.** Every handler test covers the happy path plus every precondition failure and role-gate denial listed in the spec.
12. **Contract tests are non-negotiable.** Every JSON output field listed in the spec has a presence test. Every exit code has a stability test. Every idempotency guarantee has a retry-no-op test.
13. **Always run with `-race`** when testing code that touches the database or shared state. CI enforces this, but local development should too.
14. **When writing a new handler, command, or validation rule,** write the test file in the same commit. Do not defer testing to a follow-up.
